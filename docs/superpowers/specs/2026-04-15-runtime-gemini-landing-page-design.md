# Runtime Gemini Landing Page Generation — Design Spec

**Date:** 2026-04-15
**Status:** Draft — pending user review
**Scope:** Replace the current hand-written block library composition with a runtime, Gemini-designed landing page where each section is bespoke JSX per summit. Keep Claude's ArchitectPhase for section sequencing.
**Supersedes:** The "V1 dev CLI only" path in `2026-04-15-gemini-landing-page-generation-design.md` for the runtime flow. The CLI remains useful as a dev tool but is no longer the only place Gemini runs.

---

## Context

The current live flow composes landing pages from a fixed library of ~20 hand-written React blocks. Claude's `ArchitectPhase` picks a block sequence and `CopywriterPhase` fills props. The rendered output is structurally sound but visually uniform across summits — every summit looks the same because it draws from the same block implementations.

The operator wants each summit's landing page to have a **bespoke visual design per section**, generated by Gemini 2.5 Pro at page-generation time, with per-section regeneration from Filament (Stitch-style).

## Goal

When an operator clicks **Generate Landing Page**:

1. Claude picks the section sequence (unchanged).
2. For each section, Gemini generates bespoke JSX, Tailwind-styled, using the summit's data + brand as inputs.
3. Operator previews the result in Filament, edits text/image content via form fields, regenerates individual sections they don't like, and publishes.
4. On publish, the validated JSX is SSR-rendered to static HTML once and served to visitors. Generated JSX never executes in a visitor's browser.

## Non-goals

- Dev review gate between generation and publish. Operator-approved = live.
- AI image generation. V1 uses summit-provided images only.
- Operator-facing JSX editing. Operators edit content fields and regenerate; they don't see raw code.
- Replacing Stripe/optin/checkout flows. Those stay on hand-written, trusted components that Gemini's JSX references via data-hydrate markers.
- Full-page Gemini generation (end-to-end, Gemini decides sections too). Claude's architect is already good at sequencing.

## Architecture

### Data flow

```
Operator clicks Generate
  │
  ▼
GenerateLandingPageJob (Laravel)
  │
  ├─► ArchitectPhase (Claude, unchanged) → section sequence
  │
  ├─► BlockDesignPhase (Gemini 2.5 Pro, NEW)
  │     ├─ N parallel calls, one per section
  │     ├─ Each returns { jsx, fields[] }
  │     └─ AST-validate each; on fail, retry once; on second fail, mark failed
  │
  ├─► Persist LandingPageDraft with sections[] JSON
  │
  └─► Operator opens draft in Filament
        ├─ Preview iframe (Next.js SSR of current draft state)
        ├─ Per-section: content form + "Regenerate UI" + "Regenerate with note"
        └─ Publish → SSR-to-static job → live page updated at funnel URL
```

### Components

**Reused (unchanged):**
- `ArchitectPhase` (Laravel) — picks section types + order from summit brief.
- Domain resolver middleware, funnel routing, Stripe, optin capture.
- Existing `@/components/ui/*` primitives (base-ui Accordion, Button, Card, Countdown, Input, Label, Select, Separator, Textarea).

**New:**
- `BlockDesignPhase` (Next.js side, TypeScript). One Gemini call per section. Runs in parallel. Lifts prompt-assembly logic from the existing `pnpm gen:block` CLI.
- `BlockValidator`. `@babel/parser` + AST walker. Whitelists imports, primitives, JSX shapes; rejects everything else. See _Safety_ below.
- `FieldExtractor`. Not a full AST walker — Gemini returns the editable paths itself in its output contract. `FieldExtractor` validates that the paths resolve against the JSX and clamps to expected types (string/url). If a path doesn't resolve, field is dropped.
- `LandingPagePublisher` (Laravel job updated). Calls Next.js SSR endpoint, receives static HTML + hydration runtime, stores on draft, marks draft `published`.
- `client-runtime.ts` (new, ~5 KB). Hand-written hydration layer that activates hydration markers in the SSR'd HTML.

**Removed:**
- `CopywriterPhase`. Gemini writes copy inline while designing; no separate Claude copywriting pass.

### Data model

New columns on `landing_page_drafts`:
- `sections` — JSON array. Each section: `{ id (uuid), type, jsx, fields[], status, regeneration_note, source_section_id (nullable for regen history) }`.
- `published_html` (nullable, longtext) — the SSR'd static HTML, populated on publish.
- `published_hydration_manifest` (nullable, json) — map of which hydration behaviors the HTML needs.

Existing column `blocks` (current composition blocks) is kept alongside for drafts generated under the old flow; new drafts use `sections`. Later migration cleans this up.

### Per-section data shape

```json
{
  "id": "550e8400-e29b-...",
  "type": "speakers_grid",
  "jsx": "export default function S() { return <section className='py-20'>... </section>; }",
  "fields": [
    { "path": "heading", "kind": "text", "value": "20+ World-class speakers" },
    { "path": "speakers.0.photo", "kind": "image", "value": "https://..." }
  ],
  "status": "ready",
  "regeneration_note": null,
  "source_section_id": null
}
```

## Gemini prompt shape (BlockDesignPhase)

System + multimodal user message containing:

1. Role + whitelist + output contract rules.
2. Section brief: type, purpose, position N of M.
3. Summit context: name, date, brand colors, dark/light preference, speakers, tone brief, lead product.
4. `DESIGN_SYSTEM` constant (shared with `gen:block` CLI).
5. Primitive source code (.tsx) of every `@/components/ui/*` component.
6. Reference image: one PNG from `docs/block-references/` matching the type if available.
7. Previous-section JSX excerpt (for visual flow).

Strict JSON output (no code fences):

```json
{ "jsx": "...", "fields": [{"path":"...","kind":"text","value":"..."}] }
```

Gemini returns both the code and the editable-field map. This is cheaper than writing a bulletproof AST walker that guesses "what's content" — and it gives the designer-model authorship of what's editable.

## Safety

### Gate 1 — AST validator at generation time

Every JSX string passes `@babel/parser` + a custom walker. Allowed: React import, `lucide-react` imports, `@/components/ui/*` imports, JSX elements with Tailwind classes, function declarations returning JSX, primitive literals. Rejected: any other import, embedded script or style tags, raw-HTML injection props, client hooks, network calls, dynamic code evaluation, dynamic imports, globals access through template interpolation.

On validation fail: retry once with the error fed back into the prompt. On second fail: mark the section `failed`, persist Gemini's raw output for debugging, surface in Filament with a "Regenerate" CTA.

### Gate 2 — SSR-to-static at publish

On publish, the Next.js SSR endpoint:
1. Loads each section's JSX string into an isolated module context.
2. Runs `ReactDOMServer.renderToString()` per section in a worker.
3. Concatenates HTML, inlines the hydration runtime, stores the result.
4. Writes `published_html` + `published_hydration_manifest` on the draft.

Visitors to `summit.clientsite.com/{funnel-slug}` get the pre-rendered HTML. Generated JSX never runs in visitor browsers.

On SSR failure: block publish, surface the failing section to operator, allow fix-and-regenerate without losing other sections.

### Hydration runtime

`next-app/src/runtime/client-runtime.ts` — ~5 KB hand-written. Looks for data-hydrate markers (countdown, accordion, optin-form) and wires the interactive behavior. Gemini's JSX emits markup + these attributes; the runtime is trusted, fixed, reviewed code.

## Editing UX (Filament)

Block list view (matches mockup approved during brainstorm):

- Each row shows: section number, type label, short description, inline `⟳ Regenerate UI` button, expand chevron, status badge (`ready` / `regenerating` / `failed`).

Expanded section:
1. **Content fields** — inputs bound to `fields[].path`. Saving writes back to `fields[i].value`. Renderer substitutes values at SSR time via the path map. No Gemini call.
2. **Regenerate UI with note** — textarea (max 500 chars). Submitting dispatches `RegenerateSectionJob(draftId, sectionId, note)` which calls Gemini with the note + current section JSX in the prompt. Replaces the section in place, keeping the same UUID (React key stability in preview).
3. **Danger zone** — delete, reorder, duplicate.

Top-bar draft actions:
- Preview (opens iframe rendering current draft state).
- Regenerate all (confirms first, re-runs all sections in parallel).
- Publish (disabled until every section is `ready`; triggers SSR-to-static; on success marks draft `published` and activates the funnel URL).

## Failure handling

| Failure | Behavior |
|---|---|
| Gemini returns malformed JSON | Retry once. Second fail → section status `failed`, raw output stored for debugging. |
| AST validator rejects JSX | Retry once with validator error in prompt. Second fail → `failed`. |
| Rate limit / 5xx from Gemini | Retry with exponential backoff (3 tries). Then `failed`. |
| Gemini returns `fields[].path` that doesn't resolve | Drop that field silently. |
| SSR at publish fails for a section | Block publish. Surface section + error. Operator regenerates. |
| All N sections fail | Operator sees error state, can retry generation or fall back to old composition flow (old flow still exists). |

No silent partial writes. A draft is either fully-generated + ready or visibly failed.

## Cost + latency

**Per landing page generation (~7–10 sections):**
- Gemini 2.5 Pro with image + text input: ~$0.05–0.15 per call.
- Parallel execution, wall time ~20–40s end-to-end (gated by the slowest section).
- Total cost per generation: ~$0.50–1.50.

**Per regeneration (one section):**
- One Gemini call. ~$0.05–0.15, ~15–30s.

**Publish (SSR-to-static):**
- No Gemini cost. ~2–5s CPU for a 7-section page.

## Rollout phases

1. **BlockDesignPhase + AST validator.** Stand up the per-section Gemini call behind a flag; run it against 1 section type (e.g. speakers_grid) in a spike page. Verify output quality + typecheck + SSR round-trip.
2. **Field extraction + renderer.** Server-side JSX-string → React render with field substitution. Test round-trip: generate → edit field → render reflects edit.
3. **Hydration runtime + client-side interactivity.** Hand-write `client-runtime.ts` for countdown/accordion/optin-form. Verify published HTML works in a real browser.
4. **Filament UI.** Block list + per-section regenerate + content form + publish button. Start with read-only preview, then edits, then regenerate, then publish.
5. **SSR-to-static publish pipeline.** Draft → published_html on approve. Funnel route reads from published_html.
6. **Full-page generation path.** Wire ArchitectPhase → BlockDesignPhase → draft persistence end-to-end. Run against 2–3 real summits, compare to current composition output.
7. **Feature-flag flip + deprecate old flow.** Keep old CopywriterPhase + block library available behind the flag for ~30 days, then remove.

Each phase gets its own PR, verified in isolation, merged to main. No single mega-PR.

## Open questions (post-brainstorm)

- **Image sourcing:** V1 uses summit-provided photos. Gemini references these in its JSX. Question: if a section's spec calls for an image type the summit didn't upload (e.g. "background hero image" but operator only uploaded speaker headshots), what's the fallback? Recommend: neutral gradient placeholder; flag to operator to upload.
- **Reference PNG matching:** when no reference exists for a section type (new type Claude invents), Gemini flies blind. Recommend: generate without reference on first hit; when output looks good, save it back to `docs/block-references/` to guide future generations of that type.
- **Draft history:** regenerate-a-section creates a new section record but overwrites in place today. Do operators want a per-section history slider? Parked — V1 is "regenerate replaces", no history.

## Scope limits (V1)

- One operator generation flow. No concurrent-operator conflict resolution.
- No A/B testing of generated pages. That's V2.
- No multi-language / localization. English-only.
- No animation frameworks (no Framer Motion). Tailwind animations + CSS only.
- No custom fonts beyond what's already loaded in the project.
